# HR Interview Assistant

## О проекте

**HR Interview Assistant** — это учебный AI‑проект для проведения структурированных собеседований по разным вакансиям.

Ассистент предназначен для работы в OpenAI GPTs Playground или аналогичной среде, где можно использовать системный промпт, базу знаний и набор отдельных карточек вакансий.

Основная идея проекта — отделить:
- общую логику работы ассистента;
- общую базу знаний о компании и принципах оценки кандидатов;
- отдельные карточки вакансий, которые можно добавлять, удалять и обновлять без переписывания всей системы.

## Цель проекта

Проект решает следующие задачи:
- помогает интервьюеру проводить собеседование по заранее заданной структуре;
- подсказывает релевантные вопросы по конкретной вакансии;
- поддерживает единый подход к оценке кандидатов;
- формирует итоговый структурированный отчёт по результатам интервью;
- масштабируется под разные роли за счёт отдельных файлов профилей вакансий.

## Структура проекта

Проект расположен локально по пути:

```text
C:	emp\hr-interview-assistant
```

Рекомендуемая структура:

```text
C:	emp\hr-interview-assistant
│   README.md
│   system_prompt_core.md
│   company_knowledge_base.md
│
└───job_profiles
    │   sales_soft_client_001.md
    │   info_security_specialist_001.md
    │   design_engineer_rkd_001.md
```

## Назначение файлов

### 1. `README.md`
Этот файл описывает проект, структуру папок и правила использования материалов.

### 2. `system_prompt_core.md`
Основной системный промпт ассистента.

Он задаёт:
- роль ассистента;
- порядок выбора вакансии;
- логику загрузки и сортировки карточек;
- динамическую нумерацию карточек в рамках сессии;
- сценарий интервью;
- правила формирования итогового отчёта по кандидату.

### 3. `company_knowledge_base.md`
Общая база знаний компании.

Файл нужен для описания:
- типа компании и её направлений;
- групп вакансий;
- общих принципов оценки кандидатов;
- подхода к софт‑скиллам;
- правил проведения интервью;
- ограничений и этических принципов.

### 4. Папка `job_profiles`
Папка хранит отдельные карточки вакансий.

Каждая карточка — это самостоятельный `.md`‑файл, который содержит:
- основную информацию о вакансии;
- контекст роли;
- обязанности;
- обязательные и желательные требования;
- red flags;
- банк вопросов;
- рубрику оценки;
- порог прохождения;
- финальные вопросы кандидату.

## Текущий набор карточек

На текущем этапе в проект включены следующие профили:

1. `sales_soft_client_001.md` — менеджер по продажам программного обеспечения и работе с клиентами.
2. `info_security_specialist_001.md` — специалист отдела защиты информации.
3. `design_engineer_rkd_001.md` — инженер‑конструктор (РКД, SolidWorks).

## Как работает ассистент

Логика работы проекта следующая:

1. Ассистент получает системный промпт из файла `system_prompt_core.md`.
2. Ассистент получает общую базу знаний из файла `company_knowledge_base.md`.
3. Ассистент получает набор карточек вакансий из папки `job_profiles`.
4. В начале диалога ассистент помогает выбрать нужную вакансию.
5. После выбора вакансии ассистент проводит интервью по структуре карточки.
6. В конце интервью ассистент формирует итоговый отчёт по кандидату.

## Принцип масштабирования

Проект специально собран так, чтобы его можно было расширять.

Если нужно добавить новую вакансию:
1. создаётся новый `.md`‑файл в папке `job_profiles`;
2. внутри файла оформляется карточка по той же структуре;
3. ассистент может использовать её без переписывания всей архитектуры.

Это позволяет не привязывать систему только к трём текущим ролям.

## Правила именования карточек

В проекте используются **технические имена файлов**, а не длинные человекочитаемые заголовки.

Например:
- `sales_soft_client_001.md`
- `info_security_specialist_001.md`
- `design_engineer_rkd_001.md`

При этом внутри файла должен быть полноценный заголовок карточки, например:

```markdown
## Карточка 3. Инженер‑конструктор (РКД, SolidWorks)
```

Такой подход нужен, чтобы:
- сохранить стабильные идентификаторы файлов;
- не зависеть от ручной перенумерации;
- упростить масштабирование проекта;
- позволить ассистенту динамически нумеровать карточки в рамках конкретной сессии.

## Где использовать проект

Проект можно использовать:
- в OpenAI GPTs Playground;
- в средах, где можно задать system prompt и загрузить knowledge files;
- как учебный кейс для портфолио по направлению AI‑ассистентов и автоматизации HR‑процессов.

## Как загружать материалы

Рекомендуемый порядок загрузки в GPT / Playground:

1. Вставить содержимое `system_prompt_core.md` в системный промпт.
2. Загрузить `company_knowledge_base.md` как общую базу знаний.
3. Загрузить все карточки из папки `job_profiles` как отдельные knowledge files.
4. Проверить, что ассистент корректно понимает названия вакансий и Vacancy ID.
5. Протестировать интервью минимум на 1–2 сценариях по каждой роли.

## Что стоит проверить перед сдачей

Перед финальной сдачей проекта рекомендуется проверить:
- все ли `.md`‑файлы лежат в нужных папках;
- нет ли пустых шаблонов или заглушек;
- совпадает ли структура всех карточек вакансий;
- нет ли противоречий между системным промптом и карточками;
- формирует ли ассистент итоговый отчёт в нужной структуре;
- корректно ли работает логика выбора вакансии;
- понимает ли ассистент новые карточки, если они добавлены в `job_profiles`.

## Итог

Проект **HR Interview Assistant** собран как модульная система:
- один системный промпт;
- одна общая база знаний;
- любое количество карточек вакансий;
- единая логика интервью и оценки кандидатов.

Такая архитектура делает проект удобным для демонстрации, расширения и дальнейшей доработки.
